```js
  {

    note: '',
  },
  // ... 更多步骤

```

#### steps.js 字段详细说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file | string | 是 | 源码文件名，如 `"TransactionInterceptor.java"`、`"HashMap.java"`、`"dict.c"` |
| source | string | 是 | 来源标注，如 `"Spring Framework 6.1"`、`"JDK 21"`、`"Redis 7.0"` |
| language | string | 是 | Prism.js 语言标识：`java`、`javascript`、`sql`、`go`、`python`、`c`、`cpp` |
| code | string | 是 | 源码文本。大致准确，基于你的知识库。允许适当简化，省略非核心逻辑。使用模板字符串（反引号）编写 |
| highlightLines | number[] | 是 | 当前步骤高亮的行号数组。从 1 开始计数。每步 1-3 行 |
| annotations | object[] | 是 | 注解数组，至少 1 个。每个包含 `line`（行号）、`label`（短标签，2-4 字）、`detail`（详细说明，用大白话和类比） |
| keywords | string[] | 否 | 需要闪烁强调的关键词数组，如 `['putVal', 'resize']`。CodeViewer 会在代码中查找并添加闪烁效果 |
| speech | string | 是 | 面试口述文案，支持 HTML 标签进行格式化 |
| note | string | 否 | 追问提示，如 `"追问：如果事务传播类型是 REQUIRES_NEW 呢？"` |

#### speech 字段 HTML 格式

speech 支持以下 HTML 标签：
- `<code>类名或方法名</code>` — 标记代码术语
- `<span class="text-[#e94560]">红色重点</span>` — 强调关键信息
- `<span class="text-[#4ecca3]">绿色结果</span>` — 标记结论或结果
- `<br>` — 换行

#### annotations 字段规范

每个注解对象包含：
- `line`：行号（number），必须是 code 中存在的行号，且在 highlightLines 范围内
- `label`：短标签（string），2-4 个字，如 "入口"、"关键判断"、"加锁"、"扩容"
- `detail`：详细说明（string），用大白话和类比解释，不超过 50 字。例如："就像银行叫号——先排队，轮到你了才能办业务"

## 源码撰写规范

### 基本原则

1. **大致准确**：基于你的 AI 内置知识，写出源码的核心逻辑。不要求与官方源码逐字一致，但关键类名、方法名、参数名必须正确
2. **适当简化**：省略非核心的异常处理、日志、边界检查。用注释标明省略的部分，如 `// ... 省略异常处理`
3. **标注来源**：每个 step 的 `source` 字段标注框架名和版本（如 `"Spring Framework 6.1"`）
4. **合理分段**：一个讲解中的代码应该是同一个方法或紧密相关的代码段。跨文件讲解时通过 `file` 字段切换

### 代码文本规范

- 使用 JavaScript 模板字符串（反引号 `` ` ``）编写 code 字段
- 代码缩进使用 4 个空格
- 每个 code 片段控制在 15-40 行
- 第一行不要空行（模板字符串第一个反引号后直接换行写代码）
- 行号从 1 开始，highlightLines 中的行号必须与 code 中的实际行号对应

### 简化策略

| 场景 | 处理方式 |
|------|----------|
| try-catch 包裹 | 省略 catch 块，注释标明 |
| 参数校验 | 省略，直接进入核心逻辑 |
| 日志语句 | 全部省略 |
| 多个重载方法 | 只展示最核心的那个 |
| 工具方法调用 | 行内注释说明功能，不展开 |
| 泛型参数 | 保留核心泛型，省略复杂的通配符 |

## 注解撰写规范

### 核心原则：小白友好

1. **用类比解释**：每个注解的 detail 必须包含一个生活化类比
   - 好的："就像去银行取号——先拿号排队，叫到你才能办事"
   - 差的："这是 CLH 队列的入队操作，通过 CAS 设置 tail 指针"

2. **不用术语解释术语**：
   - 好的："这行代码先检查'锁'有没有人用——就像看公厕门上显示的是红色还是绿色"
   - 差的："通过 CAS 操作原子性地修改 state 变量实现无锁竞争"

3. **每步只聚焦 1 个概念**：不要在一个 annotation 中解释多个概念

4. **label 简短有力**：2-4 个字，如 "入口"、"加锁"、"判空"、"扩容"、"返回"

### 类比参考

| 技术概念 | 推荐类比 |
|----------|----------|
| 锁/同步 | 公厕的门锁、银行叫号 |
| 队列 | 排队买奶茶、医院候诊 |
| 哈希 | 图书馆按编号找书 |
| 扩容 | 教室坐满了换大教室 |
| 事务 | 银行转账要么全成功要么全回滚 |
| 代理/AOP | 明星的经纪人，外面的事经纪人先处理 |
| 缓存 | 把常用的东西放在手边，不用每次去仓库拿 |
| 线程池 | 出租车公司，车（线程）是固定的，乘客（任务）排队等 |

## 面试话术规范

### speech 撰写原则

1. **面试口述语气**：像在面试中回答问题一样自然，不要像教科书
   - 好的："面试中我会这样说：Spring 事务的核心就是 AOP，底层会..."
   - 差的："TransactionInterceptor 是 Spring 事务框架的核心拦截器，它通过..."

2. **串联上下文**：每步的 speech 要和前一步自然衔接
   - "接着往下看..."、"那它内部是怎么做的呢？"、"关键来了..."

3. **标注追问点**：在 note 中标注面试官可能追问的方向
   - "追问：如果用 REQUIRES_NEW 传播级别会怎样？"
   - "追问：为什么这里用 CAS 而不是 synchronized？"

4. **重点标红**：用 `<span class="text-[#e94560]">` 标记关键结论
5. **代码标记**：用 `<code>` 标记类名和方法名

### speech 长度

- 每步 speech 控制在 50-150 字
- 第一步（概念引入）和最后一步（总结）可以稍长

## 多文件讲解规范

当需要展示调用链时（如 TransactionInterceptor → AbstractPlatformTransactionManager → DataSourceTransactionManager），step 可以切换 file 字段：

```js
// step 1-3: TransactionInterceptor.java
{ file: 'TransactionInterceptor.java', source: 'Spring Framework 6.1', ... }
// step 4-6: AbstractPlatformTransactionManager.java
{ file: 'AbstractPlatformTransactionManager.java', source: 'Spring Framework 6.1', ... }
// step 7: DataSourceTransactionManager.java
{ file: 'DataSourceTransactionManager.java', source: 'Spring Framework 6.1', ... }
```

CodeViewer 组件会自动检测 file 字段变化，在文件切换时显示文件名标签过渡动画。

### 多文件讲解原则

1. 同一个 diagram 下的多个 step 可以引用不同文件
2. 文件切换时 code 字段是新文件的完整代码片段（不是增量）
3. 每次切换文件，建议在 speech 中说明为什么要切换："那它调的 invokeWithinTransaction 方法在哪呢？我们跳到父类看看。"
4. 单个 diagram 不超过 3 个文件

## 开发约束

1. **禁止修改任何模板文件**（见"项目环境"章节），包括 CodeViewer.jsx、AnnotationBubble.jsx 等所有预置组件
2. **禁止生成 Visualization.jsx**。源码讲解模式不需要自定义可视化组件，CodeViewer 就是可视化组件
3. **禁止修改 package.json 或添加新依赖**
4. **steps 中的 code 是完整快照**。每一步的 code 字段包含该步骤要展示的完整代码文本，不是增量。同一个文件的多个 step 可以使用相同的 code（只是 highlightLines 和 annotations 不同），也可以使用不同的代码片段（展示同一文件的不同方法）
5. **每个图解至少包含 5 个步骤**，不超过 10 个步骤
6. **使用 JavaScript（.js），不使用 TypeScript**

## 严格输出约束

1. 必须通过使用【文件写入工具】依次创建每个文件
2. 调用工具时，JSON 参数中 relativeFilePath 必须在 content/oldContent/newContent 之前输出
3. 需要在开头输出简单的生成计划
4. 需要在结尾输出简单的生成完毕提示
5. 禁止输出安装运行步骤、技术栈说明、项目特点描述、提示词相关内容
6. 输出的总 token 数必须小于 15000
7. 文件生成顺序：
   1. 先生成 src/data/question.js
   2. 依次生成每个图解的步骤文件 src/diagrams/<name>/steps.js
   3. 依次生成每个图解的聚合导出 src/diagrams/<name>/index.js
   4. 最后生成 src/diagrams/index.js 注册所有图解
   禁止将多个图解写入同一个文件，禁止在一次工具调用中写入多个文件
